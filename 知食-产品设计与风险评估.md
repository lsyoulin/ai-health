## 一、产品定位与边界

### 1.1 产品定位（一句话）

**知食是一款「AI 慢病饮食管理工具」，通过个性化饮食影响预测 + 组合优化建议，帮助慢病患者在家完成专业营养师级的饮食决策。**

### 1.2 产品边界（做什么 / 不做什么）

| 维度 | 做 | 不做 |
|------|---|------|
| **诊断** | ❌ 不做疾病诊断 | 不做血糖诊断、不替代医生诊断 |
| **治疗** | ❌ 不做治疗方案 | 不开药、不调整用药方案 |
| **饮食建议** | ✅ 基于权威指南做饮食影响预测与组合优化 | 不做"治疗型"营养处方 |
| **药物相互作用** | ⚠️ 仅提示常见药食相互作用风险，建议咨询医生 | 不做药物调整建议 |
| **急救建议** | ⚠️ 提供急救知识科普，引导就医 | 不替代 120 急救流程 |
| **饮食记录** | ✅ 帮助记录饮食并给出影响分析 | 不做通用营养追踪 |
| **并发症预测** | ⚠️ 基于公开研究做风险趋势提示 | 不做疾病进展预测 |

**核心定位**：「**辅助决策工具**」，不是「诊断治疗工具」。这是合规的生命线。

### 1.3 法律声明（强制嵌入）

每个分析结果必须附带：
> ⚠️ 本结果基于公开营养学与慢病管理研究，由 AI 模型辅助生成，仅供参考。不能替代医生或营养师的专业建议。如有不适请立即就医。

## 三、技术实现挑战与应对方案

### 3.1 挑战一：食物识别准确率

#### 3.1.1 挑战描述

- 中餐形态复杂（炒、煮、蒸、炸），同一菜名形态差异大
- 一餐多菜组合，需要分离识别
- 分量估算难度大（一盘菜多少克）
- 用户拍照环境差异（光线、角度、距离）

#### 3.1.2 应对方案

| 阶段 | 方案 | 准确率预期 | 实施难度 |
|------|------|-----------|---------|
| **MVP 阶段** | 手动选择菜品（预置 500+ 中国常见菜）+ AI 辅助分量输入 | 95%+（因为用户已确认菜品） | 低 |
| **V2 阶段** | 接入成熟商用食物识别 API（如阿里云食物识别、百度 AI 开放平台） | 85-90% | 中 |
| **V3 阶段** | 自训练模型，基于中国家常菜数据集微调 | 90%+ | 高 |

**MVP 推荐方案**：
- 提供 500+ 中国常见菜品数据库（按菜系、食材、烹饪方式分类）
- 用户选菜后，AI 基于菜品默认分量，提供滑块调整（0.5x / 1x / 1.5x / 2x）
- 识别准确率问题通过"用户确认"绕过，不卡 MVP

### 3.2 挑战二：健康预测模型

#### 3.2.1 挑战描述

- 个体差异大（同一种食物对不同人血糖反应不同）
- 需要 Persona 化模型（糖尿病/高血压/并发症/健康人）
- 模型验证需要临床数据，普通公司难以获得

#### 3.2.2 应对方案

**三层建模法**：

```
第一层：基于指南的规则引擎（确定性）
├─ 血糖控制目标（来自 CDS 指南）
├─ 钠摄入限制（来自中国高血压指南）
├─ 食物分类规则（来自中国膳食指南）
└─ 风险阈值（来自临床研究）

第二层：基于食物成分的预测模型（数据驱动）
├─ GI × 碳水含量 → 血糖反应（基础公式）
├─ 顺序效应系数（基于 Shukla 等研究）
├─ 搭配效应系数（基于 Johnston 等研究）
└─ 个体修正因子（基于用户历史数据）

第三层：AI 个性化推理（LLM 辅助）
├─ 自然语言交互
├─ 个性化建议生成
└─ 知识卡片触发
```

**MVP 阶段**：
- 实现第一层（规则引擎）+ 第二层基础公式
- 模型验证：与公开研究数据对比，误差控制在 ±15% 以内
- 风险声明：明确"预测值仅供参考"

**商业化阶段**：
- 引入用户历史血糖数据（CGM 接入）做个体校准
- 与合作医院做小样本临床验证

### 3.3 挑战三：多变量组合优化算法

#### 3.3.1 挑战描述

- 一餐 N 种食物 × 3 种调整维度（分量/顺序/搭配）= 组合爆炸
- 需要实时响应（用户不能等超过 3 秒）
- 优化目标多（血糖+血压+饱腹感+口味可接受度）

#### 3.3.2 应对方案

| 维度 | 方案 |
|------|------|
| **搜索空间缩减** | 不做穷举，基于医学规则预筛选 Top10 候选方案 |
| **实时优化** | 使用贪心算法 + 规则约束，单次优化 <1 秒 |
| **多目标平衡** | 主目标血糖，辅目标血压与饱腹感，权重可配置 |
| **LLM 辅助** | 用 LLM 生成自然语言优化建议，提升可读性 |

### 3.4 挑战四：系统架构与扩展性

#### 3.4.1 架构演进路径

**MVP（当前）**：
- 纯前端 SPA（React + TypeScript）
- 食物数据 + 算法本地化
- 无用户系统、无历史记录持久化

**V2（6 个月）**：
- 加入轻后端（Node.js + PostgreSQL）
- 用户系统 + 历史记录持久化
- 接入 LLM API（对话教练、知识生成）

**V3（12 个月）**：
- B 端多租户架构
- 数据分析后台（B 端客户报表）
- API 开放（供保险/医院系统对接）

#### 3.4.2 技术选型

| 层 | MVP | V2 | V3 |
|----|-----|-----|-----|
| 前端 | React 18 + TypeScript + Vite | 同上 + PWA | 同上 + 小程序 |
| 后端 | 无 | Node.js + Express | + 微服务 |
| 数据库 | 无 | PostgreSQL | + Redis 缓存 |
| AI | 本地规则引擎 | + OpenAI/Claude API | + 私有模型微调 |
| 部署 | Vercel | + 云服务器 | + K8s 集群 |
| 认证 | 无 | JWT | + OAuth 2.0 |

## 四、医疗监管与合规风险评估

### 4.1 核心风险：被定性为医疗器械

#### 4.1.1 风险描述

根据《医疗器械监督管理条例》（国务院令第 739 号）与《人工智能医用软件产品分类界定指导原则》（国家药监局 2021）：

- 用于"疾病诊断、治疗、监测、缓解"的软件 → 二类/三类医疗器械
- 用于"健康管理和健康促进"的软件 → 非医疗器械

#### 4.1.2 应对方案（核心合规策略）

**产品定位严格守住"非诊断非治疗"红线**：

| 红线 | 守住策略 |
|------|---------|
| ❌ 不做疾病诊断 | 不输出"您患有 X 病"的诊断结论 |
| ❌ 不做治疗方案 | 不建议用药调整、不制定治疗方案 |
| ❌ 不替代医生 | 所有结果均声明"仅供参考，请咨询医生" |
| ❌ 不做疾病进展预测 | 并发症风险提示基于统计学，不做个体疾病进展预测 |
| ✅ 做健康管理 | 饮食影响分析、营养建议、健康促进 |
| ✅ 做知识科普 | 疾病知识教育（有来源引用） |

**用户协议与免责声明**（强制）：
1. 注册时必须同意《用户协议》与《健康信息知情同意书》
2. 每次分析结果附带免责声明
3. 设置"咨询医生"快捷入口，引导就医

#### 4.1.3 分类界定预判

基于 NMPA《人工智能医用软件产品分类界定指导原则》：
- 知食属于"健康管理机构使用的健康管理软件"，**预期用途为"对生活方式进行管理，不属于医疗器械"**
- 但若未来涉及"基于血糖数据进行个性化胰岛素剂量建议"，则可能触发二类医疗器械

**应对方案**：
- MVP 严格不接血糖数据输入（仅用户自述血糖范围）
- 未来如涉及血糖数据接入，需做 NMPA 分类界定咨询

### 4.2 广告与宣传合规

#### 4.2.1 风险

《广告法》对医疗、药品、医疗器械广告有严格限制：
- 不得涉及医疗、诊疗
- 不得保证疗效
- 不得利用医生、专家形象做证明

#### 4.2.2 应对方案

| 风险点 | 应对 |
|--------|------|
| 宣传文案 | 不用"治疗"、"治愈"、"疗效"等医疗词汇 |
| 效果承诺 | 不承诺"降低血糖 X%"，改用"研究显示该组合可降低血糖反应" |
| 医生背书 | 不使用医生形象做广告，引用医学文献需标注来源 |
| 用户证言 | 不使用"我用后血糖降了"等用户证言做广告 |

### 4.3 药食相互作用建议的边界

#### 4.3.1 风险

"药物与食物相互作用"建议涉及药物领域，可能触发监管。

#### 4.3.2 应对方案

- MVP 阶段：**不做药食相互作用建议**（删除该功能）
- V2 阶段：仅提供"华法林与维生素 K 食物"等**公开文献明确记载的常见相互作用**信息
- 不做"调整用药时间"、"调整药物剂量"等建议
- 任何药食相互作用提示都附带"请咨询医生"

## 五、数据安全与隐私合规

### 5.1 适用法规

| 法规 | 关键要求 |
|------|---------|
| 《个人信息保护法》（PIPL） | 个人信息处理需告知同意，敏感信息需单独同意 |
| 《数据安全法》 | 数据分级分类、出境安全评估 |
| 《医疗健康数据安全指南》 | 健康数据为敏感信息，需加密存储 |
| 《网络安全法》 | 网络安全等级保护（等保 2.0） |

### 5.2 健康数据合规策略

#### 5.2.1 数据分类

| 数据类型 | 敏感等级 | 示例 | 处理方式 |
|---------|---------|------|---------|
| 基本信息 | 一般 | 昵称、性别、年龄 | 加密存储 |
| **健康信息** | **敏感** | 慢病类型、血糖、血压 | 单独同意 + 加密 + 脱敏分析 |
| 饮食记录 | 敏感 | 每日饮食 | 加密存储 |
| 行为数据 | 一般 | App 使用习惯 | 匿名化分析 |

#### 5.2.2 合规措施

| 措施 | 实施方式 |
|------|---------|
| **知情同意** | 注册时弹窗展示《隐私政策》《健康信息知情同意书》，需主动勾选 |
| **最小必要原则** | 仅采集必要信息（慢病类型、年龄段），不强制要求身份证号 |
| **数据加密** | 传输 HTTPS + 存储 AES-256 加密 |
| **数据脱敏** | 分析时使用 user_id 替换，不存储姓名 |
| **数据出境** | 数据存储在中国境内服务器（阿里云/腾讯云），不出境 |
| **用户权利** | 支持数据导出、删除、注销账号 |
| **等保 2.0** | 商业化阶段过等保 2.0 三级认证 |

### 5.3 算法透明度与可解释性

#### 5.3.1 风险

《互联网信息服务算法推荐管理规定》要求算法透明、可解释。

#### 5.3.2 应对方案

- 每个预测结果附带"为什么"解释（基于哪些数据、哪些规则）
- 提供"算法说明"页面，公开核心算法原理
- 用户可查看自己的画像数据，支持修正

## 六、商业化落地风险与应对

### 6.1 风险一：B 端销售周期长

#### 6.1.1 风险

- 社区医院/健康险公司销售周期通常 6-12 个月
- 需要 POC（概念验证）+ 招标 + 合规审查
- 现金流压力

#### 6.1.2 应对方案

| 策略 | 具体行动 |
|------|---------|
| **C 端先跑通现金流** | MVP 上线即开放 C 端订阅（299/年），保证基础收入 |
| **B 端先试点后扩张** | 与 3-5 家社区医院做免费试点，3 个月后转付费 |
| **政策红利切入** | 利用国家"家庭医生签约服务"政策，作为数字化工具切入 |
| **免费试用策略** | B 端 14 天免费试用，降低决策门槛 |

### 6.2 风险二：用户付费意愿低

#### 6.2.1 风险

- 中国 C 端健康类 App 付费率普遍 <3%
- 慢病人群（尤其老年人）付费能力与意愿双低

#### 6.2.2 应对方案

| 策略 | 具体行动 |
|------|---------|
| **情感付费优先** | 重点打「为父母购买」场景，子女付费为父母用 |
| **免费增值模式** | 基础功能免费（记录+基础分析），高级功能付费（优化器+教练） |
| **年费锚定** | 299/年 vs 一次医院营养科挂号 100-300 元，AI 7×24 在线 |
| **礼品化** | 节日"健康关怀礼"，强化情感价值 |

### 6.3 风险三：竞品快速跟进

#### 6.3.1 风险

- 大厂（薄荷健康、平安好医生）可能快速跟进"预测+优化"功能
- 缺乏技术壁垒

#### 6.3.2 应对方案

| 壁垒 | 构建方式 |
|------|---------|
| **数据壁垒** | 早期积累用户饮食-健康反应关联数据，形成数据飞轮 |
| **B 端渠道壁垒** | 优先与社区医院/保险建立合作，建立替换成本 |
| **知识库壁垒** | 知识库与算法深度结合，非简单 API 调用 |
| **品牌心智** | 「知食 = 吃掉慢病」抢先占领用户心智 |

### 6.4 风险四：AI 预测不准

#### 6.4.1 风险

- 预测值与实际偏差大 → 用户失去信任
- 可能导致用户误用（如"AI 说我能吃"就多吃）

#### 6.4.2 应对方案

| 策略 | 具体行动 |
|------|---------|
| **保守预测** | 偏保守方向预测（宁可预测偏高，不让用户放心多吃） |
| **误差区间** | 显示预测区间（如"7.5-9.2 mmol/L"）而非单点值 |
| **用户校准** | 允许用户输入实际餐后血糖，反向校准模型 |
| **免责声明** | 明确"预测仅供参考，以实际测量为准" |

   - B. OpenAI/Claude —— 效果优先但合规风险
   - 推荐：A（国产），避免数据出境

4. **部署环境？**
   - A. 继续 Vercel（境外）—— 合规风险
   - B. 阿里云/腾讯云（境内）—— 推荐
   - 推荐：B

### 10.3 商业化路径

5. **第一阶段重点？**

   - 推荐：A，现金流优先
