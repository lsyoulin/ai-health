# 知食 · 产品商业方案

> 「「吃」掉慢病，让每一口都被懂」
>
> 知食 · 慢病饮食 AI 伙伴

---

## 一、产品本质：从「诊断」到「优化」

### 1.1 重新定义产品

知食**不是**一个"记录饮食"或"告诉你什么不能吃"的健康 App。
知食**是**一款基于 AI 的「饮食组合优化器」——帮慢病患者在"能吃"的前提下，找到"怎么吃最安全"的最优解。

### 1.2 核心命题转换

| 维度 | 现有产品 | 知食 |
|------|---------|------|
| 回答的问题 | "你能吃什么?"（黑名单） | "你该怎么吃?"（组合优化） |
| 思维方式 | 禁忌思维 | 可能性思维 |
| 输出 | 食物清单 | 饮食组合方案 |
| 用户感受 | 被限制 | 被赋能 |
| 价值定位 | 记录工具 | 决策助手 |

### 1.3 医学依据

| 现象 | 数据 | 来源 |
|------|------|------|
| 白米饭单独吃 GI | 83（高） | 国际血糖生成指数表 |
| 白米饭+蔬菜+蛋白质 GI | ≈55（中） | 营养学经典 |
| 先吃蔬菜再吃米饭 | 血糖峰值降 35% | Ann Nutr Metab 2013 |
| 醋搭配米饭 | 血糖反应降 20-30% | Eur J Clin Nutr |
| 同一餐组合/顺序差异 | 血糖反应差 2-3 倍 | 多项 RCT |

**结论**：同一碗米饭，因组合、顺序、搭配不同，餐后血糖差异可达 2-3 倍。这是被现有产品完全忽视的蓝海。

## 二、用户与客户：三方付费模型

### 2.1 用户分层（谁用）

| 层级 | 画像 | 规模 | 核心诉求 |
|------|------|------|---------|
| **核心用户** | 2型糖尿病+高血压患者本人 | 4.7亿慢病人群 | "这顿饭怎么吃才安全" |
| **情感用户** | 慢病患者的子女（25-45岁） | 约 2 亿 | "帮父母管理饮食，少操心" |
| **潜客** | 糖耐量异常/肥胖/代谢综合征前期 | 约 5 亿 | "预防变成慢病" |

### 2.2 客户分层（谁付费）

关键洞察：**用户≠客户**。慢病患者本人付费意愿与能力均弱，需寻找 B 端客户。

| 客户类型 | 付费动机 | 付费能力 | 触达难度 | 优先级 |
|---------|---------|---------|---------|--------|
| **B1：慢病管理服务机构/社区医院** | 提升管理效果指标、降低复诊成本 | 中（机构预算） | 中 | ★★★★★ |
| **B2：商业健康险公司** | 降低慢病并发症理赔支出 | 高 | 中高 | ★★★★★ |
| **B3：健康管理机构/体检中心** | 增值服务、客户留存 | 中高 | 中 | ★★★★ |
| **B4：药企/医疗器械（生态合作）** | 慢病生态闭环、患者数据 | 高 | 高 | ★★★ |
| **B5：终端用户（C 端付费）** | 情感付费、关怀礼物 | 低-中 | 低 | ★★★ |
| **B6：政府/医保** | 公共卫生预防、医保控费 | 极高 | 极高 | ★★（长期） |

### 2.3 付费意愿与打动逻辑

#### B1 社区医院/慢病管理机构（首要客户）

- **付费动机**：
  - 国家基本公共卫生服务要求对慢病患者进行年度随访与饮食指导
  - 现有随访饮食指导靠人工，效果差、成本高、无法持续
  - "饮食管理"是慢病随访考核的硬指标，但缺工具
- **打动逻辑**：
  - **成本对比**：1 个营养师日服务 20 人 vs 知食日服务无限人，单次随访成本从 50 元降至 5 元
  - **效果可量化**：3 个月使用后患者平均餐后血糖下降 X%、HbA1c 改善 Y%
  - **合规赋能**：为家庭医生签约服务提供数字化饮食管理工具，提升考核得分
- **付费模式**：按医生账号/年订阅，每位家庭医生 1200-2400 元/年

#### B2 商业健康险（战略客户）

- **付费动机**：
  - 慢病并发症理赔是健康险最大支出项（糖尿病并发症人均治疗费 5-15 万元）
  - 保险 "健康管理服务" 可纳入保费定价（监管允许不超过保费 18%）
  - 降低慢病并发症发生率 = 直接降低理赔支出
- **打动逻辑**：
  - **精算模型**：若 10 万慢病投保人使用知食，并发症发生率降低 15%，年节省理赔支出 1.5 亿
  - **获客杠杆**：免费赠送知食会员作为投保福利，提升投保转化率与续保率
  - **数据资产**：饮食习惯数据 + 血糖趋势数据，反哺精算定价模型
- **付费模式**：按投保人数量年费，每位投保人 30-60 元/年

#### B5 情感付费用户（C 端补充）

- **付费动机**：为父母健康购买一份"AI 营养师"，从"唠叨"转为"关怀工具"
- **打动逻辑**：
  - **情感转换**：把"妈，这个你不能吃"换为"妈，我给你分析了，加份醋拌黄瓜，你血糖反应能降 30%"
  - **价值锚定**：一年 365 元 = 一次医院营养科挂号费，但 AI 营养师 24 小时在线
  - **送礼场景**：母亲节/父亲节/重阳节/春节"健康关怀礼"
- **付费模式**：年费订阅 299-365 元/年（情感用户付费意愿最强）

### 3.3 核心功能：饮食组合优化器

**输入**：一餐的食物组合（通过拍照识别或手动选择）

**输出**：
1. **原始预测**：当前组合下餐后血糖/血压预测值
2. **优化方案**：
   - 分量调整建议（主食减 1/3 / 蛋白质加 50g）
   - 进餐顺序建议（先青菜→再肉→后主食）
   - 搭配建议（加醋拌黄瓜 / 加凉拌木耳）
3. **优化后预测**：调整后的血糖/血压预测值
4. **差值对比**：血糖反应降低 X%

**技术实现**：

```typescript
// 基于现有 healthEngine 扩展

interface MealOptimization {
  original: {
    composition: FoodItem[]
    predictedGlucose: number
    predictedBloodPressure: number
  }
  optimized: {
    adjustments: Adjustment[]
    newComposition: FoodItem[]
    predictedGlucose: number
    predictedBloodPressure: number
  }
  delta: {
    glucoseDelta: number      // 下降值
    glucoseDeltaPercent: number  // 下降比例
  }
}

// 优化算法：多变量空间搜索
function optimizeMeal(
  meal: FoodItem[],
  persona: Persona
): MealOptimization {
  // 1. 计算原始预测
  const original = predictGlucose(meal, persona)

  // 2. 生成优化候选方案
  const candidates = generateCandidates(meal, persona)
  // - 分量调整：在合理范围内搜索
  // - 顺序重排：基于顺序效应系数
  // - 搭配扩展：从搭配库推荐

  // 3. 多目标优化
  // 目标：最小化血糖峰值，同时保证饱腹感与口味可接受
  const optimized = multiObjectiveSearch(candidates, persona)

  // 4. 返回 Top3 方案
  return { original, optimized: optimized.top3, delta }
}
```

## 五、竞争壁垒

### 5.1 三层壁垒

| 层级 | 壁垒 | 形成方式 |
|------|------|---------|
| **数据壁垒** | 慢病饮食-血糖反应关联数据 | 越早积累越强，用户使用产生飞轮 |
| **算法壁垒** | 组合优化算法引擎 | 多变量优化搜索，需医学+算法能力 |
| **B 端渠道壁垒** | 社区医院/保险公司合作 | 早期 POC 积累案例与信任 |

### 5.2 难以被复制的核心能力

1. **AI 组合优化能力**：
   - 不是"识别食物+查表"——现有产品都能做
   - 是"基于个性化 Profile，在多变量空间中搜索最优饮食组合"——需要 AI 推理能力
   - 正是 TRAE 大赛强调的"AI 不可替代性"

2. **慢病 Persona 模型**：
   - 糖尿病/高血压/并发症/健康人，4 种 Persona 同一菜品产生 4 种预测结果
   - 基于医学文献与临床数据调校
   - 需要领域知识 + 算法工程能力

3. **B 端付费生态**：
   - 与社区医院/保险公司建立合作需要时间
   - 一旦建立数据对接与效果验证，替换成本极高

### 5.3 差异化定位

```
        高 AI 能力
            │
            │
            │   ★ 知食
            │
            │
 ───────────┼──────────── AI 能力
            │
            │   薄荷健康
            │
        低   │
        低 ─────────── 高
              饮食干预深度
```

- **薄荷健康/MyFitnessPal**：低 AI 能力 + 低饮食干预深度（只做记录）
- **医联/平安好医生**：低 AI 能力 + 中饮食干预深度（有营养师但人工服务）
- **知食**：高 AI 能力 + 高饮食干预深度（AI 预测+组合优化）

## 六、市场切入点策略

### 6.1 GTM 三阶段

**Phase 1（0-6 月）：MVP + B1 试点**

- **目标**：验证 B1 社区医院付费意愿
- **动作**：
  1. 完成 MVP（拍照识别+预测+组合优化器）
  2. 联系 5-10 家社区医院试点，免费提供 3 个月
  3. 收集患者使用数据与效果数据
  3. 产出《知食 × XX 社区医院慢病饮食管理效果报告》
- **成功指标**：
  - 5 家试点医院
  - 500 名患者使用
  - 3 个月餐后血糖平均下降 10%

**Phase 2（6-18 月）：B1 规模化 + B2 启动**

- **目标**：B1 跑通付费模式，B2 启动首个保险合作
- **动作**：
  1. 基于试点数据，向 200 家社区医院销售
  2. 与 1 家商业健康险公司启动联合 POC
  3. 积累 1 万名 C 端用户
- **成功指标**：
  - 50 家 B1 付费客户
  - 1 家 B2 合作签约
  - 1 万 C 端用户

**Phase 3（18-36 月）：B2 规模化 + 生态拓展**

- **目标**：B2 成为主要营收来源
- **动作**：
  算法持续优化
  3-5 家保险合作
  药企/食材商合作启动
- **成功指标**：
  - 3 家 B2 合作
  - 10 万+ C 爆发用户
  - B2 营收占比 >50%

### 6.2 优先客户画像

**B1 首选客户**：
- 一二线城市社区医院/社区卫生服务中心
- 已开展家庭医生签约服务
- 慢病管理基础较好、考核压力大的机构
- 院长/主任对数字化工具接受度高

**B2 首选客户**：
- 已有慢病保险产品的公司（如众安、平安、人保健康）
- 已搭建健康管理平台但缺饮食管理工具
- 注重健康险创新与服务差异化

## 七、风险与应对

| 风险类型 | 具体风险 | 应对策略 |
|---------|---------|---------|
| **技术风险** | 拍照识别准确率不足 | MVP 阶段先支持手动选择食物，识别逐步优化 |
| **医学风险** | 预测不准导致用户误用 | 明确"辅助决策"定位，非"医疗诊断"声明 |
| **监管风险** | 被定性为医疗器械 | 定位为"健康管理工具"，不涉及诊断与治疗建议 |
| **商业化风险** | B 端销售周期长 | 同时启动 C 端情感付费，保证现金流 |
| **竞争风险** | 大厂快速跟进 | 抢占 B 端渠道与数据飞轮，建立先发优势 |
